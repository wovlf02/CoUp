## 1. 일반 사용자 기능 전체 분석

### 1.1 인증 및 계정 관리

#### 사용자 가능 기능
- **회원가입**: 이메일/비밀번호, Google OAuth, GitHub OAuth
- **로그인/로그아웃**: 세션 관리, JWT 토큰 기반
- **프로필 관리**: 이름, 아바타, 자기소개 수정
- **비밀번호 변경**: 현재 비밀번호 확인 후 변경
- **계정 설정**: 알림 설정, 테마 설정, 개인정보 관리

#### 관리자 개입 필요 사항
- ❗ **계정 정지**: 이용 약관 위반 사용자 제재
- ❗ **정지 해제**: 정지된 계정 복구 (사유 검토 후)
- ❗ **강제 비밀번호 재설정**: 보안 위협 발견 시
- ❗ **탈퇴 계정 복구**: 삭제 요청 취소 (30일 이내)
- ❗ **의심 활동 모니터링**: 다중 계정, 봇 활동 탐지
- ❗ **역할 변경**: 일반 사용자 → 관리자 승격/강등

#### 데이터베이스 관련
```prisma
model User {
  status         UserStatus @default(ACTIVE)  // 관리자가 관리
  suspendedUntil DateTime?                    // 정지 기간
  suspendReason  String?                      // 정지 사유
  role           UserRole @default(USER)      // 역할 관리
}

enum UserStatus {
  ACTIVE      // 정상 활동
  SUSPENDED   // 정지됨 (관리자 제재)
  DELETED     // 삭제 요청 (30일 후 완전 삭제)
}

enum UserRole {
  USER           // 일반 사용자
  ADMIN          // 일반 관리자
  SYSTEM_ADMIN   // 시스템 관리자 (최고 권한)
}
```

### 1.2 스터디 관리

#### 사용자 가능 기능
- **스터디 생성**: 이름, 설명, 카테고리, 태그, 최대 인원, 공개/비공개 설정
- **스터디 탐색**: 카테고리별 검색, 키워드 검색, 필터링
- **스터디 가입**: 공개 스터디 즉시 가입, 비공개 스터디 신청
- **스터디 탈퇴**: 자발적 탈퇴 (OWNER는 탈퇴 전 권한 위임 필요)
- **멤버 관리** (OWNER/ADMIN만):
  - 가입 승인/거절
  - 멤버 추방 (KICK)
  - 역할 변경 (ADMIN ↔ MEMBER)
- **스터디 설정** (OWNER만):
  - 기본 정보 수정
  - 모집 중/모집 완료 토글
  - 스터디 삭제

#### 관리자 개입 필요 사항
- ❗ **불건전 스터디 모니터링**: 불법/음란/사행성 콘텐츠 감지
- ❗ **신고된 스터디 검토**: 사용자 신고 접수 시 검토 및 조치
- ❗ **강제 스터디 삭제**: 이용 약관 위반 스터디 삭제
- ❗ **스터디 복구**: 실수로 삭제된 스터디 복구 요청 처리
- ❗ **스터디 강제 비공개 전환**: 문제 발생 시 임시 비공개 처리
- ❗ **OWNER 권한 위임**: OWNER 장기 부재 시 권한 이전
- ❗ **스터디 통계 및 품질 관리**: 저품질 스터디 개선 권고
- ❗ **추천 스터디 선정**: 우수 스터디 큐레이션

#### 데이터베이스 관련
```prisma
model Study {
  id          String  @id @default(cuid())
  ownerId     String
  isPublic    Boolean @default(true)     // 관리자가 강제 변경 가능
  isRecruiting Boolean @default(true)
  rating      Float?  @default(0)        // 관리자가 조정 가능
  
  // 관리자 메타 정보 추가 필요
  adminNotes      String?  // 관리자 메모
  featuredUntil   DateTime? // 추천 스터디 종료일
  warningCount    Int @default(0) // 경고 횟수
}
```

### 1.3 커뮤니케이션 (채팅, 화상통화)

#### 사용자 가능 기능
- **실시간 채팅**: Socket.IO 기반 그룹 채팅
- **파일 공유**: 채팅에 파일 첨부
- **메시지 수정/삭제**: 본인 메시지만 가능
- **화상 통화**: WebRTC 기반 화면 공유

#### 관리자 개입 필요 사항
- ❗ **부적절한 메시지 모니터링**: 욕설, 혐오 표현, 스팸 감지
- ❗ **신고된 메시지 검토**: 사용자 신고 시 메시지 내용 확인
- ❗ **강제 메시지 삭제**: 이용 약관 위반 메시지 삭제
- ❗ **채팅 기능 제한**: 특정 사용자의 채팅 권한 일시 정지
- ❗ **스팸 발송자 제재**: 반복적인 스팸 메시지 발송자 차단
- ❗ **대화 내용 감사**: 법적 요청 시 대화 내용 제출

#### 데이터베이스 관련
```prisma
model Message {
  id      String  @id @default(cuid())
  content String  @db.Text
  
  // 관리자 메타 정보 추가 필요
  isDeleted       Boolean @default(false)
  deletedBy       String?  // 관리자 ID
  deletedReason   String?
  reportCount     Int @default(0)
}
```

### 1.4 콘텐츠 관리 (공지, 파일)

#### 사용자 가능 기능
- **공지사항 작성** (OWNER/ADMIN): 제목, 내용, 중요도 설정
- **공지 고정/해제**: 중요 공지 상단 고정
- **파일 업로드**: 학습 자료 공유
- **파일 다운로드**: 공유된 파일 다운로드
- **파일 삭제**: 업로더만 삭제 가능

#### 관리자 개입 필요 사항
- ❗ **저작권 침해 파일 삭제**: DMCA 신고 접수 시 파일 삭제
- ❗ **불법 파일 모니터링**: 음란물, 해적판 콘텐츠 감지
- ❗ **악성 파일 차단**: 바이러스, 멀웨어 파일 자동 차단
- ❗ **스토리지 용량 관리**: 저장 공간 초과 스터디 제한
- ❗ **파일 복구**: 실수로 삭제된 파일 복구 요청 처리

- ❗ **이벤트 캘린더**: 플랫폼 전체 이벤트 (챌린지, 워크숍 등)

### 1.6 신고 및 모더레이션

#### 사용자 가능 기능
- **사용자 신고**: 스팸, 괴롭힘, 부적절한 콘텐츠 신고
- **스터디 신고**: 불건전 스터디 신고
- **메시지 신고**: 부적절한 메시지 신고
- **신고 이력 확인**: 본인이 신고한 내역 조회

#### 관리자 개입 필요 사항
- ❗ **신고 접수 및 검토**: 모든 신고 내용 확인
- ❗ **신고 처리**: 승인/거절 결정
- ❗ **제재 조치**: 경고, 정지, 기능 제한, 영구 차단
- ❗ **신고 남용 감지**: 허위 신고 반복 사용자 제재
- ❗ **신고 통계 분석**: 신고 트렌드 파악 및 예방 조치

#### 데이터베이스 관련
```prisma
model Report {
  id         String     @id @default(cuid())
  reporterId String
  targetType TargetType  // USER, STUDY, MESSAGE
  targetId   String
  type       ReportType  // SPAM, HARASSMENT, INAPPROPRIATE, COPYRIGHT, OTHER
  reason     String     @db.Text
  status     ReportStatus @default(PENDING)
  priority   Priority   @default(MEDIUM)
  
  // 처리 정보
  processedBy String?    // 관리자 ID
  processedAt DateTime?
  resolution  String?    // 처리 결과
}

enum ReportStatus {
  PENDING      // 대기 중
  IN_PROGRESS  // 검토 중
  RESOLVED     // 처리 완료
  REJECTED     // 기각됨
}
```

## 2. 관리자가 개입해야 하는 영역

### 2.1 사용자 관리 (User Management)

#### 핵심 관리 작업
1. **사용자 모니터링**
   - 신규 가입자 추적 (일/주/월 통계)
   - 활성 사용자 vs 휴면 사용자 분석
   - 이탈 사용자 패턴 분석

2. **계정 상태 관리**
   - 정상(ACTIVE) / 정지(SUSPENDED) / 삭제(DELETED) 상태 전환
   - 정지 기간 설정 (1일, 3일, 7일, 30일, 영구)
   - 정지 사유 기록 및 공지

3. **제재 시스템**
   - 경고 (WARNING): 1차 위반 시 경고 메시지
   - 일시 정지 (SUSPEND): 반복 위반 시 계정 일시 정지
   - 기능 제한 (RESTRICT): 특정 기능만 차단 (채팅, 스터디 생성 등)
   - 영구 차단 (BAN): 심각한 위반 시 영구 차단

4. **사용자 검색 및 필터링**
   - 이메일, 이름, ID로 검색
   - 역할별 필터 (USER, ADMIN, SYSTEM_ADMIN)
   - 상태별 필터 (활성, 정지, 삭제)
   - 가입일 범위 필터

5. **사용자 상세 정보 조회**
   - 기본 정보 (이메일, 이름, 가입일)
   - 활동 이력 (마지막 로그인, 참여 스터디 수)
   - 제재 이력 (경고 횟수, 정지 이력)
   - 신고 이력 (신고한 횟수, 신고당한 횟수)

---

### 2.2 스터디 관리 (Study Management)

#### 핵심 관리 작업
1. **스터디 모니터링**
   - 전체 스터디 통계 (활성, 종료, 삭제)
   - 카테고리별 분포 분석
   - 평균 멤버 수, 활동도 분석

2. **품질 관리**
   - 저품질 스터디 탐지 (활동 없음, 멤버 0명 등)
   - 우수 스터디 선정 및 추천
   - 스터디 평점 관리

3. **스터디 제재**
   - 강제 비공개 전환
   - 스터디 삭제
   - 스터디 생성 제한 (특정 사용자)

4. **스터디 지원**
   - OWNER 권한 위임 요청 처리
   - 스터디 복구 요청 처리
   - 분쟁 중재 (멤버 간 갈등)

---

### 2.3 신고 관리 (Report Management)

#### 핵심 관리 작업
1. **신고 접수 및 분류**
   - 긴급도 우선순위 설정 (LOW, MEDIUM, HIGH, URGENT)
   - 신고 유형별 분류 (SPAM, HARASSMENT, INAPPROPRIATE, COPYRIGHT, OTHER)
   - 담당자 할당

2. **신고 검토 프로세스**
   ```
   [신고 접수] → [증거 확인] → [대상자 확인] → [판단] → [조치] → [완료]
   ```
   - 신고 내용 정확성 확인
   - 증거 자료 검토 (스크린샷, 메시지 로그 등)
   - 피신고자 활동 이력 조회
   - 위반 여부 판단
   - 적절한 제재 조치 결정
   - 신고자 및 피신고자에게 결과 통보

3. **제재 조치 실행**
   - 경고 메시지 발송
   - 계정 정지 (기간 설정)
   - 콘텐츠 삭제 (메시지, 파일, 스터디)
   - 기능 제한 (채팅, 스터디 생성 등)

4. **신고 통계 분석**
   - 일/주/월별 신고 건수
   - 신고 유형별 분포
   - 처리 시간 평균
   - 반복 신고자/피신고자 추적

---

### 2.4 콘텐츠 모더레이션 (Content Moderation)

#### 핵심 관리 작업
1. **자동 필터링**
   - 욕설/비속어 자동 감지
   - 스팸 패턴 감지 (반복 메시지, 광고 링크)
   - 악성 파일 바이러스 스캔
   - AI 기반 부적절한 이미지 탐지

2. **수동 검토**
   - 신고된 메시지 내용 확인
   - 저작권 침해 파일 검토
   - 불건전 스터디 설명 검토

3. **콘텐츠 조치**
   - 메시지 삭제 및 로그 기록
   - 파일 삭제 및 업로더 제재
   - 공지사항 강제 수정/삭제

---

### 2.5 시스템 설정 및 운영 (System Management)

#### 핵심 관리 작업
1. **시스템 설정 관리**
   - 최대 스터디 멤버 수 제한 설정
   - 파일 업로드 용량 제한 설정
   - 신규 가입 승인 여부 설정
   - 메일 발송 제한 설정

2. **서비스 제한 설정**
   - 사용자별 스터디 생성 제한 (예: 최대 5개)
   - 사용자별 스터디 참여 제한 (예: 최대 10개)
   - API 요청 제한 (Rate Limiting)

3. **백업 및 복구**
   - 데이터베이스 정기 백업
   - 삭제된 콘텐츠 복구
   - 시스템 롤백

4. **로그 관리**
   - 관리자 작업 로그 기록
   - 사용자 활동 로그 분석
   - 보안 이벤트 로그 모니터링

---

### 2.6 분석 및 리포팅 (Analytics & Reporting)

#### 핵심 관리 작업
1. **사용자 통계**
   - 일일 활성 사용자 (DAU)
   - 월간 활성 사용자 (MAU)
   - 신규 가입자 추이
   - 이탈율 분석

2. **스터디 통계**
   - 활성 스터디 수
   - 카테고리별 인기도
   - 평균 스터디 수명
   - 멤버 충족률

3. **서비스 건강도**
   - 평균 응답 시간
   - 에러 발생률
   - 서버 리소스 사용률
   - 데이터베이스 성능

4. **비즈니스 지표**
   - 사용자 참여도 (Engagement)
   - 리텐션 (Retention)
   - 전환율 (Conversion)
   - 수익 (향후 프리미엄 기능 시)

## 3. 관리자 역할별 권한 정의

### 3.1 역할 계층 구조

```
SYSTEM_ADMIN (시스템 관리자)
    ↓ 모든 권한
ADMIN (일반 관리자)
    ↓ 제한된 권한
USER (일반 사용자)
```

---

### 3.2 SYSTEM_ADMIN (시스템 관리자) - 최고 권한

#### 전용 권한
- ✅ **모든 ADMIN 권한 포함**
- ✅ **시스템 설정 변경**: 전역 설정 수정
- ✅ **관리자 임명/해임**: USER → ADMIN 승격, ADMIN → USER 강등
- ✅ **데이터베이스 직접 관리**: 직접 쿼리 실행, 데이터 수정
- ✅ **백업 및 복구**: 시스템 전체 백업, 복구
- ✅ **감사 로그 접근**: 모든 관리자 활동 로그 조회
- ✅ **API 키 관리**: 외부 서비스 API 키 설정
- ✅ **서버 관리**: 서버 재시작, 배포, 모니터링
- ✅ **요금제 설정** (향후): 프리미엄 기능 활성화

#### 접근 가능 페이지
- `/admin/*` - 모든 관리자 페이지
- `/admin/system-settings` - 시스템 설정 (전용)
- `/admin/admins` - 관리자 관리 (전용)
- `/admin/database` - 데이터베이스 관리 (전용)
- `/admin/logs` - 감사 로그 (전용)

---

### 3.3 ADMIN (일반 관리자) - 제한된 권한

#### 권한 범위
- ✅ **사용자 관리**: 조회, 정지, 기능 제한 (삭제 불가)
- ✅ **스터디 관리**: 조회, 비공개 전환, 삭제
- ✅ **신고 처리**: 신고 검토, 승인/거절, 제재 조치
- ✅ **콘텐츠 모더레이션**: 메시지/파일 삭제, 공지 수정
- ✅ **통계 조회**: 대시보드, 분석 리포트 열람
- ❌ **시스템 설정 변경 불가**
- ❌ **관리자 임명 불가**
- ❌ **데이터베이스 직접 접근 불가**

#### 접근 가능 페이지
- `/admin/dashboard` - 관리자 대시보드
- `/admin/users` - 사용자 관리
- `/admin/studies` - 스터디 관리
- `/admin/reports` - 신고 관리
- `/admin/moderation` - 콘텐츠 검열
- `/admin/analytics` - 통계 분석

---

## 4. 관리자-사용자 상호작용 시나리오

### 시나리오 1: 신고 접수 및 처리

```
1. [사용자 A] 사용자 B의 스팸 메시지 신고
   ↓
2. [시스템] 신고 접수, 관리자에게 알림
   ↓
3. [관리자] 신고 내용 확인 및 증거 검토
   ↓
4. [관리자] 사용자 B의 메시지 히스토리 조회
   ↓
5. [관리자] 판단:
   - 신고 승인 → 사용자 B에게 경고 발송
   - 반복 위반 시 → 3일 채팅 기능 정지
   ↓
6. [시스템] 사용자 A, B에게 처리 결과 통보
```

---

### 시나리오 2: 악성 스터디 발견 및 제재

```
1. [관리자] 일일 스터디 모니터링 중 의심 스터디 발견
   ↓
2. [관리자] 스터디 상세 정보 확인
   - 설명: 불건전 내용 포함
   - 멤버: 대부분 신규 가입자
   - 활동: 과도한 외부 링크 공유
   ↓
3. [관리자] 조치:
   - 즉시 스터디 비공개 전환
   - OWNER에게 경고 메시지 발송
   - 48시간 내 수정 요청
   ↓
4. [OWNER] 수정 안 함 → [관리자] 스터디 강제 삭제
   또는
   [OWNER] 수정 완료 → [관리자] 재검토 후 공개 승인
```

---

### 시나리오 3: 계정 정지 및 복구

```
1. [사용자 C] 반복적인 규칙 위반 (3회 경고)
   ↓
2. [관리자] 7일 계정 정지 결정
   - suspendedUntil: 2025-12-04
   - suspendReason: "반복적인 욕설 사용"
   ↓
3. [시스템] 사용자 C 로그인 시도 시 정지 안내 페이지 표시
   ↓
4. [사용자 C] 7일 후 자동 정지 해제
   또는
   [사용자 C] 정지 해제 요청 → [관리자] 검토 후 조기 해제
```

---

### 시나리오 4: 시스템 전체 공지

```
1. [SYSTEM_ADMIN] 긴급 점검 공지 필요
   ↓
2. [SYSTEM_ADMIN] 전체 사용자 알림 생성
   - 제목: "긴급 점검 안내"
   - 내용: "2025-11-28 02:00 ~ 04:00 시스템 점검"
   - 타입: SYSTEM (시스템 알림)
   ↓
3. [시스템] 모든 활성 사용자에게 알림 발송
   - 앱 푸시 알림
   - 이메일 발송
   - 사이트 배너 표시
```

---

### 시나리오 5: 신규 관리자 임명

```
1. [SYSTEM_ADMIN] 신뢰할 수 있는 사용자 D를 관리자로 임명하기로 결정
   ↓
2. [SYSTEM_ADMIN] 사용자 D 조회
   - 활동 이력: 우수
   - 신고 이력: 없음
   - 가입일: 6개월 이상
   ↓
3. [SYSTEM_ADMIN] 역할 변경
   - role: USER → ADMIN
   ↓
4. [시스템] 사용자 D에게 관리자 임명 통보
   - 관리자 페이지 접근 권한 부여
   - 관리자 가이드 제공
   ↓
5. [SYSTEM_ADMIN] 감사 로그 기록
   - action: ROLE_CHANGE
   - before: USER
   - after: ADMIN
   - reason: "신뢰할 수 있는 활동 이력"
```

## 6. 결론

CoUp 플랫폼의 일반 사용자 기능을 체계적으로 분석한 결과, **관리자가 개입해야 하는 6개 핵심 영역**을 도출했습니다:

1. **사용자 관리**: 계정 상태, 제재, 역할 관리
2. **스터디 관리**: 품질 관리, 제재, 지원
3. **신고 관리**: 신고 검토, 처리, 제재 조치
4. **콘텐츠 모더레이션**: 부적절한 콘텐츠 감지 및 삭제
5. **시스템 설정**: 전역 설정, 제한 값 관리
6. **분석 및 리포팅**: 통계, 건강도 모니터링

이러한 영역을 효과적으로 관리하기 위해 **2단계 권한 시스템**(ADMIN, SYSTEM_ADMIN)을 구축하며, 각 역할의 권한을 명확히 구분하여 책임 소재를 분명히 합니다.

다음 문서에서는 **일반적인 웹사이트 관리자 시스템 모범 사례**를 조사하여 CoUp에 적용 가능한 추가 기능을 도출합니다.
