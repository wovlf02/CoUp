### 슬라이드 3: 프로젝트 개요 (40초)

**[열정적으로, 핵심 가치 강조]**

> CoUp은 Cooperate와 Up의 합성어로, 스터디 그룹의 생성부터 관리, 협업까지 모든 것을 지원하는 **올인원 플랫폼**입니다.
> 
> CoUp이 제공하는 네 가지 핵심 가치를 말씀드리겠습니다.
> 
> **첫째, All-in-One 통합 환경**입니다. 채팅, 화상 통화, 파일 공유, 캘린더, 할일 관리까지 스터디에 필요한 모든 기능을 하나의 플랫폼에서 제공합니다.
> 
> **둘째, 스마트 매칭**입니다. 카테고리별 검색과 필터링을 통해 개인의 학습 목표에 맞는 최적의 그룹을 찾을 수 있습니다.
> 
> **셋째, 데이터 기반 동기부여**입니다. 출석, 진도, 활동 분석을 통해 지속 가능한 학습 환경을 조성합니다.
> 
> **넷째, 건강한 커뮤니티**입니다. 함께 성장하는 학습 문화를 만들어갑니다.
> 
> 이 프로젝트는 11월 4일부터 11월 22일까지, 총 19일간 개발되었습니다.

### 슬라이드 6: 기술 스택 (40초)

**[전문성을 보여주며, 간결하게]**

> 이제 CoUp을 구현하기 위해 사용한 기술 스택을 소개하겠습니다.
> 
> **먼저 프론트엔드와 백엔드는 통합되어 있습니다.** Next.js 16과 React 19를 기반으로 하며, Prisma 6로 데이터베이스를 관리하고, NextAuth.js로 인증을 처리합니다. UI는 Tailwind CSS로 구성했으며, Socket.IO Client와 React Query를 활용해 실시간 데이터와 서버 상태를 관리합니다.
> 
> **시그널링 서버는 분리형 아키텍처**로 구성했습니다. Node.js와 Express.js 기반이며, Socket.IO와 Redis Adapter를 사용해 실시간 통신과 확장성을 확보했습니다.
> 
> **데이터베이스와 인프라**는 PostgreSQL 15와 Redis 7을 사용하며, Docker로 컨테이너화하고, AWS S3로 파일을 저장합니다.
> 
> **실시간 통신**은 WebRTC를 활용한 P2P 화상 통화와 Socket.IO 기반 WebSocket으로 구현했으며, STUN과 TURN 서버를 통해 네트워크 연결을 지원합니다.

### 슬라이드 7: 아키텍처 - 전체 구조 (30초)

**[다이어그램을 가리키며 설명]**

> CoUp의 전체 시스템 아키텍처를 보여드리겠습니다.
> 
> **브라우저 클라이언트**는 HTTPS를 통해 **Next.js 앱**과 통신하며, Next.js는 프론트엔드를 서버 사이드 렌더링하고 API Routes를 통해 백엔드 기능을 제공합니다.
> 
> **실시간 통신**은 별도의 **시그널링 서버**가 담당합니다. 브라우저는 WebSocket, 즉 Socket.IO를 통해 시그널링 서버와 연결되며, 시그널링 서버는 내부 API를 통해 Next.js와 통신합니다.
> 
> **데이터 계층**은 PostgreSQL이 주 데이터베이스로 사용되며, Redis는 캐싱과 Pub/Sub 메시지 브로커 역할을 합니다. Next.js는 Prisma ORM을 통해 PostgreSQL과 통신하고, 시그널링 서버는 Redis와 직접 연결됩니다.
> 
> **파일 저장**은 AWS S3를 사용합니다.
> 
> 이러한 구조를 통해 각 컴포넌트가 독립적으로 확장 가능하도록 설계했습니다.

### 슬라이드 8: 아키텍처 - 설계 원칙 (30초)

**[전문성 어필, 각 원칙의 의미 강조]**

> 이러한 아키텍처를 설계하면서 **네 가지 핵심 원칙**을 적용했습니다.
> 
> **첫째, 분리형 아키텍처**입니다. Next.js는 HTTP와 REST API를 담당하고, 시그널링 서버는 WebSocket만 전담합니다. 이를 통해 각 서비스를 독립적으로 확장할 수 있습니다.
> 
> **둘째, 실시간 통신**입니다. Socket.IO와 Redis Adapter를 결합하여 다중 서버 환경에서도 실시간 메시지를 전달할 수 있으며, 수평 확장이 가능합니다.
> 
> **셋째, 마이크로서비스 지향**입니다. 느슨한 결합을 유지하고 Redis Pub/Sub 메시징을 활용하여 서비스 간 의존성을 최소화했습니다.

> **세 번째는 스터디 탐색 페이지**입니다. 카테고리별로 스터디를 검색하고 필터링할 수 있으며, 각 스터디의 정보를 카드 형태로 보여줍니다.
> 
> **네 번째는 스터디 생성 페이지**입니다. 스터디 이름, 설명, 카테고리, 최대 인원 등을 입력하여 새로운 스터디를 만들 수 있습니다.
> 
> **다섯 번째는 채팅 화면**입니다. 실시간으로 메시지를 주고받을 수 있으며, 무한 스크롤로 이전 대화 내역을 볼 수 있습니다.
> 
> **여섯 번째는 공지사항**입니다. Markdown으로 작성된 공지를 깔끔하게 렌더링하여 보여줍니다.
> 
> 이 외에도 화상 통화, 파일 공유, 캘린더, 할일, 마이페이지, 관리자 페이지 등 **총 23개의 화면이 구현되었습니다.**

**[포인트]**

### 슬라이드 12: 핵심 로직 설명 (1분)

**[다이어그램을 가리키며, 기술적 깊이 제시]**

> CoUp의 핵심 기술 중 세 가지를 깊이 있게 설명드리겠습니다.
> 
> **첫째, WebRTC 화상 통화 구현**입니다. 화면의 시퀀스 다이어그램을 보시면, 클라이언트 A가 시그널링 서버를 통해 방에 입장하면, 서버는 클라이언트 B에게 새로운 사용자가 들어왔다고 알립니다. 그다음 클라이언트 A가 Offer를 생성하고 시그널링 서버를 통해 클라이언트 B에게 전달합니다. 클라이언트 B는 Answer를 생성하여 다시 A에게 보냅니다. 이후 ICE Candidate를 교환하면서 두 클라이언트 간의 P2P 연결이 완성됩니다. 이렇게 시그널링 서버는 연결을 중개하는 역할만 하고, 실제 영상과 음성 데이터는 클라이언트 간 직접 전송됩니다.
> 
> **둘째, 실시간 채팅 아키텍처**입니다. 클라이언트가 Socket.IO를 통해 메시지를 전송하면, 시그널링 서버가 이를 Redis Pub/Sub으로 발행하고, 데이터베이스에 저장합니다. 이후 Redis를 구독하고 있는 모든 서버가 메시지를 수신하여 연결된 클라이언트들에게 브로드캐스트합니다. 이를 통해 여러 시그널링 서버가 있어도 일관된 메시지 전달이 가능합니다.
> 
> **셋째, 권한 기반 접근 제어**입니다. 모든 API 요청은 미들웨어를 거쳐 세션을 확인하고, 역할을 체크한 후 API 핸들러로 전달됩니다. 이를 통해 관리자만 공지를 작성하거나 멤버를 강퇴하는 등의 권한 제어가 가능합니다.

**[포인트]**
- 다이어그램의 흐름을 차근차근 설명
- 기술 용어(Offer, Answer, ICE, Pub/Sub)를 정확히 발음
- 각 단계의 목적을 명확히 전달

**[자신감 있게, 여유롭게]**

> 이상으로 CoUp 프로젝트 발표를 마치겠습니다.
>

>

**[Q&A 대응 팁]**

**예상 질문 1**: "WebRTC P2P 방식의 한계는 무엇인가요?"

> 좋은 질문입니다. P2P Mesh 방식은 최대 6명 정도까지는 안정적이지만, 그 이상이 되면 각 참여자가 모든 다른 참여자와 연결을 유지해야 하므로 대역폭 문제가 발생합니다. 향후 SFU(Selective Forwarding Unit) 방식을 도입하여 더 많은 참여자를 지원할 계획입니다.

**예상 질문 2**: "기존 플랫폼(Zoom, 카카오톡)과의 차별점은?"

> CoUp의 가장 큰 차별점은 '통합'입니다. Zoom은 화상만, 카카오톡은 채팅만, Notion은 문서만 제공하지만, CoUp은 이 모든 기능을 하나의 플랫폼에서 제공하여 스터디 관리의 효율성을 극대화합니다. 또한 무료로 제공된다는 점도 큰 장점입니다.

**예상 질문 3**: "확장성(scalability) 측면에서 어떻게 설계했나요?"

> 시그널링 서버를 Next.js와 분리하고, Redis Pub/Sub를 활용하여 수평 확장이 가능하도록 설계했습니다. 시그널링 서버는 Docker 컨테이너로 실행되어 Auto Scaling이 가능하며, Next.js는 Vercel의 서버리스 환경에서 자동으로 확장됩니다.

**예상 질문 4**: "데이터베이스 병목 현상은 어떻게 해결하나요?"

> PostgreSQL의 인덱스 최적화와 Redis 캐싱을 활용합니다. 자주 조회되는 데이터는 Redis에 캐싱하고, 실시간 메시지는 Redis Pub/Sub를 거쳐 비동기로 DB에 저장하여 부하를 분산시킵니다.

**예상 질문 5**: "보안은 어떻게 보장하나요?"

> NextAuth.js를 통한 인증, JWT 기반 세션 관리, 그리고 역할 기반 접근 제어(RBAC)를 구현했습니다. 또한 파일 업로드 시 크기와 확장자를 검증하고, AWS S3에 Pre-signed URL을 사용하여 보안을 강화했습니다.

**예상 질문 6**: "개발 기간이 19일인데, 가장 어려웠던 부분은?"

> WebRTC 시그널링 서버와 Next.js를 연동하는 부분이 가장 도전적이었습니다. 두 서버 간의 인증과 메시지 전달을 구현하는 과정에서 많은 시행착오가 있었지만, Redis Pub/Sub와 내부 API를 통해 해결했습니다.
