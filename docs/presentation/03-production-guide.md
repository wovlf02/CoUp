## 📋 카테고리별 예상 질문

### 1. 기술적 질문 (Technical Questions)

#### Q1: WebRTC P2P 방식의 한계는 무엇인가요?

**답변**:
> 좋은 질문입니다. P2P Mesh 방식은 최대 6명 정도까지는 안정적이지만, 참여자가 늘어나면 각자가 모든 다른 참여자와 개별 연결을 유지해야 하므로 **대역폭 문제**가 발생합니다. 예를 들어 10명이 참여하면 각 참여자가 9개의 연결을 유지해야 하므로 네트워크 부하가 급증합니다.
> 
> 이를 해결하기 위해 향후 **SFU(Selective Forwarding Unit) 방식**을 도입할 계획입니다. SFU는 중앙 서버가 스트림을 중계하는 방식으로, 각 클라이언트는 서버와만 연결하면 되므로 더 많은 참여자를 지원할 수 있습니다.

**핵심 키워드**: 대역폭, Mesh vs SFU, 확장성

---

#### Q2: Next.js와 시그널링 서버를 분리한 이유는?

**답변**:
> Next.js는 HTTP 기반의 REST API와 SSR/CSR을 담당하고, 시그널링 서버는 WebSocket 기반의 실시간 통신만 전담하도록 **관심사를 분리**했습니다.
> 
> 이렇게 하면 세 가지 장점이 있습니다. 첫째, **독립적인 확장**이 가능합니다. 화상 통화 사용자가 급증하면 시그널링 서버만 스케일 아웃하면 됩니다. 둘째, **기술 스택 최적화**가 가능합니다. Next.js는 서버리스로, 시그널링은 WebSocket을 잘 지원하는 Node.js로 구성할 수 있습니다. 셋째, **장애 격리**가 가능합니다. 한쪽에 문제가 생겨도 다른 쪽은 정상 작동합니다.

**핵심 키워드**: 관심사 분리, 마이크로서비스, 확장성

---

#### Q3: 데이터베이스 병목 현상은 어떻게 해결하나요?

**답변**:
> 세 가지 전략을 사용합니다.
> 
> **첫째, 인덱스 최적화**입니다. 자주 조회되는 컬럼(예: 스터디 카테고리, 사용자 ID)에 인덱스를 생성하여 쿼리 속도를 향상시켰습니다.
> 
> **둘째, Redis 캐싱**입니다. 스터디 목록, 사용자 프로필, 읽지 않은 알림 개수 같은 자주 조회되는 데이터는 Redis에 캐싱하여 DB 부하를 줄였습니다.
> 
> **셋째, 비동기 처리**입니다. 채팅 메시지는 Redis Pub/Sub를 통해 즉시 전달하고, DB 저장은 비동기로 처리하여 실시간성을 유지하면서도 DB 부하를 분산시켰습니다.

**핵심 키워드**: 인덱싱, 캐싱, 비동기

#### Q4: 보안은 어떻게 보장하나요?

**답변**:
> 여러 레이어에서 보안을 구현했습니다.
> 
> **인증 레이어**에서는 NextAuth.js로 Google과 GitHub OAuth를 구현하고, JWT 기반 세션을 사용합니다.
> 
> **인가 레이어**에서는 역할 기반 접근 제어(RBAC)를 구현하여 OWNER, ADMIN, MEMBER별로 권한을 구분했습니다. 예를 들어 공지사항은 관리자만 작성할 수 있습니다.
> 
> **데이터 보호**를 위해 파일 업로드 시 크기(최대 50MB)와 MIME 타입을 검증하고, AWS S3 Pre-signed URL로 직접 접근을 차단했습니다.
> 
> **통신 보안**을 위해 프로덕션 환경에서는 HTTPS와 WSS(WebSocket Secure)를 사용합니다.

**핵심 키워드**: OAuth, JWT, RBAC, HTTPS

#### Q5: Redis는 어떤 역할을 하나요?

**답변**:
> Redis는 세 가지 역할을 합니다.
> 
> **첫째, 캐시 저장소**입니다. 자주 조회되는 데이터를 임시 저장하여 DB 부하를 줄입니다.
> 
> **둘째, Pub/Sub 메시지 브로커**입니다. 채팅 메시지나 알림을 여러 시그널링 서버 인스턴스에 동시에 전달하는 역할을 합니다. 이를 통해 수평 확장 시에도 모든 사용자가 동일한 메시지를 받을 수 있습니다.
> 
> **셋째, Socket.IO Adapter**로 사용됩니다. Redis Adapter를 통해 여러 서버 간의 WebSocket 연결을 관리합니다.

**핵심 키워드**: 캐싱, Pub/Sub, 수평 확장

#### Q6: Prisma를 ORM으로 선택한 이유는?

**답변**:
> 세 가지 이유입니다.
> 
> **첫째, 타입 안전성**입니다. Prisma는 스키마에서 자동으로 TypeScript 타입을 생성하여 컴파일 시점에 오류를 잡을 수 있습니다.
> 
> **둘째, 직관적인 API**입니다. SQL을 직접 작성하는 것보다 `prisma.user.findMany()`같은 메소드가 더 읽기 쉽고 유지보수하기 좋습니다.
> 
> **셋째, 마이그레이션 관리**가 편리합니다. `prisma migrate`로 스키마 변경을 자동으로 추적하고 DB에 반영할 수 있습니다.

**핵심 키워드**: 타입 안전성, 생산성, 마이그레이션

### 4. 비즈니스 및 확장 질문 (Business & Scalability)

#### Q15: 수익 모델은 무엇인가요?

**답변**:

#### Q16: 얼마나 많은 사용자를 지원할 수 있나요?

> 
> 예상으로는 **동시 접속자 1만 명** 정도까지는 큰 문제없이 지원 가능할 것으로 보이며, 그 이상은 추가 최적화가 필요합니다.

**핵심 키워드**: 수평 확장, 서버리스, 1만 동시 접속
